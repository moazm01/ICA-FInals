Introduction

What is the Tomasulo Algorithm?

03 / 12

Developed by Robert Tomasulo at IBM in 1967, this hardware algorithm allows a processor to execute instructions as soon as their data operands become available, even if earlier instructions are still stalled. It is a foundational concept for superscalar processors.

Primary Problem Solved

In-order pipelines halt execution completely when an instruction stalls waiting for memory or computation. This stalls all subsequent, independent instructions. Tomasulo bypasses this wall entirely by scheduling execution dynamically.

⚠
Eliminates False Dependencies
Bypasses name conflicts to focus only on true data flow
The Pipeline Hazard Types

RAW — Read After Write

True Data Dependency

An instruction requires a value not yet computed. This dependency cannot be avoided; instructions must stall until the data is produced.

WAR — Write After Read

Name/Anti-Dependency

An instruction writes to a register before a prior instruction has finished reading it. Resolved by renaming.

WAW — Write After Write

Name/Output Dependency

An instruction writes to a register before a prior write finishes. Resolved by renaming.

The Algorithm

Three Primary Simulation Phases

04 / 12
Phase 1

1. Issue

Fetch the instruction from the queue. If a Reservation Station (a temporary buffer next to the execution units) is free, send the instruction there.

This dispatch process renames destination registers dynamically to eliminate name dependency bottlenecks.

if station_free: dispatch()
Phase 2

2. Execute

If the instruction's operands are ready (available in registers or computed), execute it.

If operands are not yet ready, monitor the Common Data Bus (CDB) continuously, waiting for the active values to drop from the bus.

while not_ready: listen_cdb()
Phase 3

3. Write Result

Broadcast the calculated value onto the Common Data Bus (CDB).

This allows both waiting reservation stations and the register file to grab the value simultaneously in the same clock cycle, resolving dependencies instantly.

cdb.broadcast(tag, value)
The Core Mechanics

Dynamic renaming & Bus Communication

05 / 12

How the Simulator Eliminates Bottlenecks

Tomasulo's design operates as a distributed system inside the CPU. Rather than relying on a single register file, the logic is spread out directly to the execution units.

Dynamic Data Path

When an instruction issues, it immediately copies the current data value (or tag) from the registers to the station buffer. This frees up the register file to accept updates from later instructions.

🏷️ Hardware Renaming

It eliminates False Dependencies (WAR - Write After Read, and WAW - Write After Write) by using reservation stations to dynamically rename registers. Registers point to the station tag, not a fixed memory index.

📡 The Common Data Bus (CDB)

A single bus that connects all execution units directly to the reservation stations, bypassing the main register file to eliminate write-port bottlenecks and register file read latency.

By routing data values dynamically, we allow instructions to execute in the exact order data becomes available, yielding high instruction concurrency.

Instruction Data Class
@dataclass class Instruction: id: int # unique index opcode: str # "ADD.D", "MUL.D"... dest: str # destination register src1: str # source operand 1 src2: str # source operand 2 # Cycle tracking (None = stage not reached) issue_cycle: Optional[int] = None start_exec: Optional[int] = None end_exec: Optional[int] = None write_back_cycle: Optional[int] = None
Reservation Station Data Class
@dataclass class ReservationStation: name: str # "Add1", "Mult2"... busy: bool = False op: Optional[str] = None vj, vk: Optional[float] = None # ready values qj, qk: Optional[str] = None # pending tags busy_cycles_left: Optional[int] = None

Data Flow Logic

  • Vj / Vk — hold ready operand values. Execution fires immediately when both are non-None.
  • Qj / Qk — hold active tag strings. The station waits for the named RS to write to the CDB.
  • busy_cycles_left — counts down each clock cycle. Reaching 0 triggers the Write Back stage.

Execution Latencies

ADD / SUB 2 cycles
MUL 4 cycles
DIV 12 cycles
L.D / S.D 2 cycles

💡 Latencies are user-configurable. Changing them in the UI updates the simulation data models instantly.

Why Reverse Pipeline Order?

Executing Write Back → Execute → Issue prevents a newly issued instruction from completing in the same cycle it is dispatched. This matches physical logic propagation delays.

Real-World Applications

💻
Modern CPUs

Intel, AMD, Apple Silicon

Every main consumer processor shipping today uses a Tomasulo-derived scheme. Physical execution engines utilize Reorder Buffers (ROBs) and Reservation Station pools to keep arithmetic units packed.

Intel Core: 97 reservation stations
Apple M-Series: 600+ ROB entries
🤖
AI Accelerators

GPU & NPU Engines

NVIDIA Tensor cores and Google TPUs leverage scoreboard and tag scheduling to feed massive vector grids. This ensures execution pipeline efficiency during high-latency memory reads.

NVIDIA Warp Dispatchers: dynamic scoreboards
Google TPU: vector register flow-forwarding
🔧
Compilers & JITs

Instruction Schedulers

LLVM optimization passes construct explicit data dependence graphs (mimicking hardware station tags) to statically schedule assembly operations, minimizing runtime pipeline stalls.

LLVM DAG: data-dependence scheduling
V8 Engine: JIT loop dependencies renaming
💡

Tomasulo's concept of decoupling names from values remains the baseline framework for high-throughput computing.
