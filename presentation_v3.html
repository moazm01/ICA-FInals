The Sequential Wall: Why Simple Processors Stall

Standard processors execute in program order. If an instruction waits for a slow resource, the entire pipeline stalls.

Data Hazard: Read After Write (RAW) forces waiting.
Blocked Pipelines: Independent commands block behind stalls.
// In-Order Pipeline Stall Example
Cycle 1-4: MUL.D F0, F2, F4 (Takes 4 cycles)
Cycle 5-6: ADD.D F6, F0, F8 (Stalls waiting for F0)
PIPELINE STALL: ADD.D block ⛔ 2 cycles lost
Cycle 7-8: SUB.D F10, F12, F14 (Blocked in queue, even though independent)
* A slow multiplier stalls the entire in-order CPU.

Under the Hood: Python Engine Design

To accurately model hardware parallelism in sequential Python, the simulator uses a Reverse Pipeline Order execution.

Why Reverse Order? Prevents 0-cycle timing bypasses. Guarantees 1-cycle minimum latency, honoring clock propagation bounds.
